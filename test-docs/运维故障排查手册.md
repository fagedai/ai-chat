# 运维故障排查手册

## 一、告警响应流程

### 1.1 告警分级

P0（严重）：核心服务不可用，影响全部用户。响应时间 ≤ 5 分钟，需立即通知值班负责人和技术总监。典型场景：数据库宕机、主节点网络中断、支付通道全挂。

P1（高）：核心功能降级，影响部分用户。响应时间 ≤ 15 分钟。典型场景：单个微服务 OOM、Redis 主从切换失败、CDN 回源率飙升到 50% 以上。

P2（中）：非核心功能异常。响应时间 ≤ 1 小时。典型场景：通知服务延迟、报表生成失败、日志采集断流。

P3（低）：体验性问题。响应时间 ≤ 4 小时。典型场景：页面加载慢（但未超时）、个别接口偶发 500、前端样式错乱。

### 1.2 通用排查步骤

无论哪个级别的告警，排查流程遵循"五步定位法"：

**第一步：确认影响范围**
- 检查 Grafana 全局 Dashboard，确认是全局故障还是局部故障
- 查看受影响的用户比例（通过 Sentry 错误率 / 总请求数）
- 确认是否有近期发布（GitLab CI 最近 1 小时的部署记录）

**第二步：检查基础设施**
- 阿里云监控：ECS 实例健康状态、网络流入流出、磁盘使用率
- K8s 事件：`kubectl get events --sort-by=.metadata.creationTimestamp`
- 节点资源：`kubectl top nodes` 检查 CPU/内存是否接近 Limit

**第三步：检查应用层**
- 查看应用日志：`kubectl logs -f deployment/app-service --tail=100`
- 检查错误堆栈，关注：OutOfMemoryError、ConnectionRefused、TimeoutException
- 查看最近的代码变更：`git log --oneline -10`

**第四步：检查依赖服务**
- 数据库：`SELECT count(*) FROM pg_stat_activity;` 查看连接数
- Redis：`redis-cli info` 检查内存使用和连接数
- 消息队列：Kafka Consumer Lag 监控（lag > 10000 说明消费积压）

**第五步：恢复与复盘**
- 优先恢复服务（重启、扩容、回滚），再排查根因
- 故障恢复后 24 小时内完成 RCA（Root Cause Analysis）文档
- RCA 文档需包含：时间线、根因、影响范围、修复措施、预防方案

## 二、常见故障场景与解决方案

### 2.1 服务 OOM（内存溢出）

**现象**：Pod 频繁重启，`kubectl describe pod` 显示 `OOMKilled`
**根因分析**：
- 最常见：Node.js 堆内存泄漏。默认堆上限约 1.7GB（V8 引擎），当内存使用持续增长不释放时会触发 OOM
- 次常见：一次性加载了过大的数据集（如导出 10 万条记录的 Excel）

**排查命令**：
```bash
# 查看 Pod 内存使用趋势
kubectl top pod app-service-xxx --containers

# 查看最近重启记录
kubectl get pods -o wide | grep RESTARTS

# 进入容器获取堆快照（需提前开启 --inspect 端口）
node --inspect app.js
```

**解决方案**：
- 短期：增加 Pod 的 memory limit（从 2Gi 调至 4Gi）
- 长期：使用 clinic.js 或 Chrome DevTools 分析堆快照，定位内存泄漏点
- 典型泄漏源：未关闭的数据库连接、全局变量累积、事件监听器未移除

### 2.2 数据库连接池耗尽

**现象**：接口响应时间从 50ms 飙升至 5000ms+，日志中出现 `E-9001` 错误码
**根因分析**：
- 慢查询长时间占用连接（最常见）
- 应用代码中未正确释放连接（忘记 await 或 try-catch 中遗漏释放）
- 突发流量超过连接池上限

**排查命令**：
```sql
-- 查看当前活跃连接
SELECT pid, usename, application_name, state, query_start, query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY query_start ASC;

-- 杀掉长时间运行的查询（谨慎使用）
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'active'
AND query_start < NOW() - INTERVAL '30 seconds';

-- 查看连接池使用情况（PgBouncer）
SHOW POOLS;
```

**解决方案**：
- 紧急：重启应用服务释放所有连接
- 短期：增加 PgBouncer 的 default_pool_size（从 20 调至 40）
- 长期：为所有查询设置 statement_timeout = 30s，添加慢查询告警

### 2.3 Redis 缓存雪崩

**现象**：数据库 CPU 突然飙升至 90%+，接口延迟增加 3-5 倍
**根因分析**：大量缓存 key 同时过期，请求全部穿透到数据库

**排查命令**：
```bash
# 查看 Redis 内存和 key 统计
redis-cli info memory
redis-cli dbsize

# 查看 key 的 TTL 分布（采样 1000 个 key）
redis-cli --scan --pattern 'product:*' | head -1000 | xargs -I {} redis-cli ttl {}

# 查看慢查询
redis-cli slowlog get 10
```

**解决方案**：
- 紧急：临时将热点数据手动写入 Redis（通过脚本批量 SET）
- 短期：为所有 key 的 TTL 增加随机偏移（±20%），避免同时过期
- 长期：引入多级缓存（L1 本地缓存 + L2 Redis），本地缓存兜底

### 2.4 Kafka 消息积压

**现象**：下游业务（如通知、报表）数据延迟严重，Kafka Consumer Lag 持续增长
**根因分析**：
- 消费者处理速度跟不上（最常见：消费者中有同步 HTTP 调用导致超时）
- 消费者挂了但未触发 Rebalance
- 生产者突发流量（如大促活动）

**排查命令**：
```bash
# 查看消费者组 Lag
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group notification-consumer

# 查看 Topic 分区分布
kafka-topics.sh --bootstrap-server kafka:9092 \
  --describe --topic user-events

# 查看消费者心跳
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group notification-consumer --members
```

**解决方案**：
- 紧急：增加消费者实例数（从 3 扩至 6），触发 Rebalance
- 短期：优化消费者中的慢操作（异步化 HTTP 调用、批量写入数据库）
- 长期：设置合理的 max.poll.records（从 500 降至 100），避免单次处理过多消息超时

## 三、监控与告警配置

### 3.1 核心监控指标

| 指标 | 告警阈值 | 采集工具 |
|------|---------|---------|
| 接口 P99 延迟 | > 1000ms | Prometheus + Histogram |
| 错误率（5xx） | > 1% | Nginx access log |
| 数据库活跃连接 | > 80% pool size | pg_stat_activity |
| Redis 内存使用 | > 80% maxmemory | redis-cli info |
| Kafka Consumer Lag | > 10000 | kafka-consumer-groups |
| Pod 重启次数 | > 2 次/小时 | kube-state-metrics |
| 磁盘使用率 | > 85% | node_exporter |

### 3.2 告警通知渠道

- P0/P1：钉钉群（值班群） + 电话通知（阿里云语音告警）
- P2：钉钉群（运维群）
- P3：邮件通知（每周汇总）
