# 平台架构设计与技术选型文档

## 第一章：系统总体架构

### 1.1 架构演进历程

我们的平台经历了三个阶段的架构演进。第一阶段是单体架构（Monolithic），所有功能模块打包在一个 Node.js 进程中，使用 Express 框架提供 RESTful API。这个阶段的优点是开发效率高，部署简单，适合团队规模在 5 人以下、日活用户在 1 万以内的场景。但随着业务增长，单体架构的弊端逐渐显现：单个模块的性能瓶颈会拖垮整个系统，代码库超过 10 万行后协作效率急剧下降。

第二阶段我们采用了 SOA（面向服务架构），将核心业务拆分为用户服务、订单服务、支付服务、通知服务四个独立微服务。服务间通过 RabbitMQ 进行异步通信，同步调用则使用 gRPC。这个阶段解决了单体架构的扩展性问题，但引入了分布式系统的复杂性：网络延迟、数据一致性、服务发现等问题接踵而至。团队为此额外投入了 2 名 SRE 工程师来维护基础设施。

第三阶段（当前阶段）我们采用了"务实的微服务"策略：只将高并发模块（API 网关、实时通知、文件处理）独立部署，核心业务逻辑回收到一个模块化的单体应用中（Modular Monolith）。这个策略将运维成本降低了 40%，同时保持了关键模块的独立扩展能力。

### 1.2 当前技术栈

前端：React 18 + TypeScript + Next.js 14（App Router），状态管理使用 Zustand，UI 组件库基于 Ant Design 5.x 定制。构建工具使用 Turbopack（开发环境）和 Webpack 5（生产环境），打包体积通过 Tree Shaking 和动态导入控制在 200KB（gzip 后）以内。

后端：Node.js 20 LTS + Fastify 框架，ORM 使用 Drizzle（替代了之前的 Prisma，查询性能提升约 30%）。数据库使用 PostgreSQL 16，连接池通过 PgBouncer 管理，读写分离采用一主两从架构。缓存层使用 Redis 7（Cluster 模式），热点数据命中率保持在 85% 以上。

搜索引擎：Elasticsearch 8.x，主要用于全文检索和日志分析。索引策略按业务场景分为两类：商品搜索使用 IK 分词器 + 拼音分词，日志搜索使用标准分词器。每日索引量约 500 万条文档，集群配置为 3 节点、每个索引 2 个分片。

消息队列：Kafka 3.6（替代了之前的 RabbitMQ），主要承载三类消息：用户行为事件（日均 2000 万条）、订单状态变更（日均 50 万条）、系统告警（按需触发）。消费者组按业务线隔离，确保一个业务线的消费延迟不会影响其他业务。

### 1.3 部署架构

生产环境部署在阿里云 ACK（容器服务 Kubernetes 版），共 12 个 ECS 节点（8 台计算型 c7.xlarge + 4 台内存型 r7.xlarge）。核心服务部署为 Deployment（副本数 3），非核心服务部署为 StatefulSet（副本数 2）。

流量入口使用阿里云 ALB（应用型负载均衡），配置了以下策略：
- 灰度发布：新版本先承接 10% 流量，观察 30 分钟无异常后逐步提升至 100%
- 限流：单 IP 每秒最多 50 次请求，超过后返回 429 Too Many Requests
- WAF 防护：阿里云 Web 应用防火墙，拦截 SQL 注入、XSS、CSRF 等常见攻击

CI/CD 流程使用 GitLab CI，从代码提交到生产部署平均耗时 8 分钟。流水线包含：代码静态检查（ESLint + Prettier）→ 单元测试（Vitest，覆盖率要求 ≥ 80%）→ 集成测试（Playwright）→ Docker 镜像构建 → 推送至 ACR（阿里云容器镜像服务）→ Helm 部署至 K8s。

## 第二章：性能优化实践

### 2.1 数据库优化

慢查询治理是我们每个 Sprint 的固定任务。通过 pg_stat_statements 插件监控 TOP 10 慢查询，典型的优化手段包括：

1. **索引优化**：对高频查询的 WHERE 条件建立复合索引。例如订单列表页的查询 `WHERE user_id = ? AND status IN (?, ?) ORDER BY created_at DESC`，建立了 `(user_id, created_at DESC)` 复合索引后，P99 延迟从 800ms 降至 50ms。

2. **查询改写**：将 `SELECT *` 改为只查询需要的字段，减少 IO 开销。一个典型案例是用户列表页，原先查询了 45 个字段，改为只查 8 个展示字段后，接口响应时间降低了 60%。

3. **分页优化**：将 OFFSET 分页改为游标分页（Cursor-based Pagination）。当 OFFSET 超过 10000 时，查询时间会线性增长。游标分页通过 `WHERE id > last_id LIMIT 20` 实现，无论翻到第几页都保持恒定性能。

4. **连接池调优**：PgBouncer 配置为 Transaction 模式，max_client_conn 设为 200，default_pool_size 设为 20。通过监控发现，约 15% 的连接处于 idle in transaction 状态，通过在应用层设置 statement_timeout = 30s 解决了长事务占用连接的问题。

### 2.2 缓存策略

我们采用多级缓存架构：

- L1（进程内缓存）：使用 Node.js 的 Map 存储热点配置数据（如系统参数、功能开关），TTL 为 5 分钟。优点是零延迟，缺点是进程间不共享。
- L2（Redis 缓存）：存储用户会话、权限信息、商品详情等，TTL 根据数据类型设置（会话 24h，权限 1h，商品 10min）。
- L3（CDN 缓存）：静态资源和公开 API 响应通过阿里云 CDN 缓存，TTL 为 1 小时。

缓存穿透防护：对于不存在的 key，缓存空值并设置较短 TTL（30 秒），防止恶意请求打穿数据库。布隆过滤器在商品搜索场景中使用，预加载了全量商品 ID，拦截无效查询。

缓存雪崩防护：TTL 在基础值上加随机偏移（±20%），避免大量 key 同时过期。同时配置了 Redis 的 maxmemory-policy 为 allkeys-lru，内存不足时自动淘汰最久未访问的数据。

### 2.3 接口性能指标

当前核心接口的 P99 延迟：
- 用户登录：120ms（含 JWT 签发）
- 商品列表：45ms（含分页，命中缓存时 12ms）
- 商品详情：30ms（命中缓存时 8ms）
- 订单创建：350ms（含库存扣减、支付单创建）
- 搜索接口：200ms（Elasticsearch 查询 + 结果聚合）

SLA 承诺：核心接口可用性 ≥ 99.95%，P99 延迟 ≤ 500ms。月度故障时间预算为 21.6 分钟（30天 × 24h × 0.05%）。

## 第三章：安全与合规

### 3.1 数据安全

敏感数据处理遵循以下原则：
- 密码存储：使用 bcrypt（cost factor = 12），单次哈希耗时约 250ms，有效防止暴力破解
- 手机号、身份证号：使用 AES-256-GCM 加密存储，密钥通过 KMS（密钥管理服务）管理，每季度自动轮换
- 日志脱敏：所有包含 PII（个人身份信息）的字段在写入日志前自动脱敏，手机号显示为 138****5678

### 3.2 接口安全

API 安全策略包含多层防护：
- 认证：JWT Token + Refresh Token 双令牌机制，Access Token 有效期 2 小时，Refresh Token 有效期 30 天
- 授权：RBAC（基于角色的访问控制），权限粒度支持到接口级别，共定义了 15 个角色、200+ 个权限点
- 防重放：关键写操作（支付、转账）要求携带 nonce 和 timestamp，服务端校验时间窗口为 ±5 分钟
- 限流：基于令牌桶算法，普通接口 100 req/min，支付接口 10 req/min

### 3.3 合规要求

系统已通过以下安全认证：
- 等保三级（信息系统安全等级保护）
- ISO 27001（信息安全管理体系）
- SOC 2 Type II（服务组织控制审计）

数据驻留要求：所有用户数据存储在中国大陆境内（阿里云杭州区域），跨境传输需经过数据脱敏和法务审批。
